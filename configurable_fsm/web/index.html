Understanding the Architecture

The project replaces four separate, hard-coded VHDL state machines with one reusable FSM engine — the generic_fsm.vhd. Behavior is stored not in logic gates but in a look-up table (ROM). Swap the ROM data and the same chip becomes a different controller.

Think of it like a CPU: the processor core stays the same; only the program (data in memory) changes what it does. Here, the FSM core stays the same; only the ROM table changes what it controls.

3-stage pipeline: event captured → ROM fetched → state updated. Outputs are valid 2 clock cycles after an event fires.

What is an FSM?

A Finite State Machine moves between a fixed set of states based on inputs. At any moment it occupies exactly one state, and an event causes a transition while producing outputs. Every traffic light, vending machine, and protocol handler is an FSM at its core.

🔥

The Hard-Coded Problem

Traditional FSM design means writing a unique VHDL entity for every controller — traffic lights hard-wired inside the traffic light module, elevator states inside another — duplicating the same clock/reset boilerplate everywhere. Changing behavior means rewriting logic, which introduces bugs.

✅

Our Solution: Write Once

One generic_fsm.vhd core handles any application. It simply looks up (state + event) in a ROM table, extracts the next state and output, and updates registers on the clock edge. Behavior lives in the table, not the logic.

🧩

Configure Many Times

config_rom.vhd stores four independent lookup tables — one per application — selectable by a 2-bit config_id. The same FSM core simultaneously supports all four controllers. Adding a 5th means adding a ROM section, not rewriting logic.

Compare & Contrast: Hard-Coded vs ROM-Driven FSM

This section contrasts the traditional approach (writing a separate VHDL state machine per controller) with the configurable ROM-driven approach used in this project. Understanding these tradeoffs is key to understanding why the architecture is designed the way it is.

Dimension Hard-Coded FSM ROM-Driven (This Project)
Consistency Each FSM is a separate VHDL entity written independently. Different developers write slightly different reset logic, output encoding, or pipeline structures — leading to subtle inconsistencies and difficult cross-controller debugging. One generic core, tested once. All four controllers share identical pipeline timing, reset behavior, and interrupt mechanism. If a bug is found and fixed in generic_fsm.vhd, all four controllers benefit instantly.
Feasibility Simple and familiar — VHDL CASE statements map cleanly to state diagrams. No indirection. Small FSMs (≤8 states) synthesize efficiently. The approach breaks down for large or frequently-changing state spaces. Requires ROM infrastructure up front (config_rom.vhd, address encoding, pipeline stages). Slightly higher initial complexity. Becomes vastly more feasible for 3+ controllers since new controllers only require ROM data — no new logic.
Efficiency Single-cycle transitions (no pipeline overhead). Optimal for FPGA area when only one controller is needed. Logic is directly synthesized from CASE — no ROM inference needed. 2-cycle pipeline latency per transition (ROM read + state update). ROM occupies BRAM or LUT-RAM. However: one core instead of four means lower total area when multiple controllers are instantiated on the same chip.
Modifiability Changing behavior = editing VHDL, re-synthesizing, re-placing-and-routing. Risk of introducing bugs. Must regression-test the entire entity. Changing behavior = updating ROM data (one constant). Logic path unchanged. Regression tests for the FSM core remain valid. Only ROM entries need re-verification.
Scalability Adding a 5th controller means copy-pasting an existing FSM entity, renaming it, and rewriting the state logic — O(N) work per controller. Adding a 5th controller means: new ROM section (data) + thin wrapper adapter (30–50 lines) + increment config_id range. Core logic untouched. O(1) growth for the FSM engine.
Testability Each FSM needs its own testbench. Errors in one don't inform others. Timing peculiarities vary per entity. Test generic_fsm.vhd once with tb_fsm.vhd — pipeline, interrupts, hold-state, reset all covered. Application-specific tests (tb_traffic_light etc.) only need to verify ROM data correctness, not FSM logic.
Choose Hard-Coded when…
  • Building exactly one, small, permanent controller (≤6 states)
  • Timing is ultra-critical and every cycle matters
  • Team has no time to learn the ROM-driven abstraction
Choose ROM-Driven when…
  • Building 2+ controllers sharing the same FPGA
  • Behavior may change after initial deployment
  • Consistency and reusability matter more than single-cycle latency
  • Long-term maintainability is a priority

Generic FSM Core

The heart of the entire project — generic_fsm.vhd

The Generic FSM is a table-driven, ROM-configured, reusable state machine engine. Rather than encoding state transitions as logic gates (VHDL CASE statements), this core reads its next-state and output values from a ROM every clock cycle. The same hardware block — the same entity, the same port map — becomes a traffic light controller, a vending machine, an elevator, or a serial receiver just by pointing it at a different section of the configuration ROM. This makes generic_fsm.vhd the single most important file in the project: it is instantiated once but handles all four applications simultaneously.

The 3-Stage Pipeline
Stage 1 Event Capture

On rising clock edge, if fsm_busy=0 and non-zero event_code arrives, capture into event_code_reg and assert fsm_busy_i='1'.

→ event_code_reg, fsm_busy_i
→
Stage 2 ROM Capture

Address {config_id, state, event_code_reg} is already at the ROM. This stage registers the ROM output into config_data_p2.

→ config_data_p2, fsm_busy_p2
→
Stage 3 State Update

Decode ROM word: extract next_state[28:24], output_action[23:8], control flags. Apply priority: Interrupt > Hold > Normal.

→ current_state, output_action_reg

State Transition Priority

1 · Interrupt (Highest)

When event_code = interrupt_event AND ROM has interrupt_en='1', FSM forces to state 0 (IDLE) immediately regardless of any other condition.

2 · Hold State

When ROM bit[0] = hold_state='1', FSM stays in current state. Event acknowledged (output_valid pulses) but current_state does not change.

3 · Normal Transition

Default path: current_state ← ROM[28:24]. Next state is whatever is programmed in the ROM lookup table for this (state, event) pair.

Common Questions
Why use a 2-stage pipeline instead of single-cycle?▼
ROM lookup takes a full clock cycle. Without the pipeline, config_data would not be stable when the state register samples it. The 2-stage design gives the ROM a full period to produce valid output before Stage 3 consumes it — satisfying setup time without increasing clock period constraints.
Why does hold_state matter?▼
Without a hold_state entry, unhandled events hit the others=>0 ROM default — meaning next_state=0 (IDLE). The FSM would silently reset on any unmapped event. hold_state lets the designer say "this event is acknowledged and intentionally ignored in this state."
How does the ROM address calculation work exactly?▼
config_addr = config_id[16:15] & state[14:10] & event_code[9:0]. For traffic light: config_id="00", state=RED("00001"), event=TIMER("0000000100"): addr = 00_00001_0000000100 = 0x0404. ROM entry at 0x0404 maps to GREEN state.

FSM Wrappers

The thin adapters that connect the real world to the generic FSM core

A wrapper is a small VHDL entity that sits between the physical hardware signals (buttons, sensors, LEDs, motors) and the generic FSM core. It has exactly two jobs: translate physical inputs → a 10-bit event code the FSM understands, and translate the FSM's 16-bit output_action → physical output signals. Nothing more. All the state-machine intelligence lives in the core and the ROM — the wrapper is just a translator.

4 wrappers in this project · ~50–80 lines each · Zero state logic inside · One per application
Where the Wrapper Lives

The wrapper (highlighted in cyan) is the only component that knows about the specific application. The FSM core and ROM are completely reusable.

The Wrapper's Two Jobs
①
Input Decoder — Physical Signals → event_code

The wrapper's input decoder watches all physical input pins every clock cycle. When it detects an active signal (and the FSM core is not busy), it encodes that signal into a 10-bit one-hot event code and sends it to the generic FSM core. Only one event fires per cycle — the priority encoder decides which if multiple signals arrive simultaneously.

Rule 1 Only fires when fsm_busy = '0' — the pipeline gate
Rule 2 Priority encoder: one event per cycle (highest priority wins)
Rule 3 Edge detection on level signals (prevents double-firing)
Rule 4 Interrupt events bypass the normal priority and go directly to the FSM interrupt port
②
Output Decoder — output_action → Physical Signals

When the FSM core completes a state transition, it asserts output_valid and puts a 16-bit bitmask on output_action. The wrapper's output decoder reads each bit and drives the corresponding physical output signal. This is a purely combinational decode — no logic, no conditions, just bit assignment.

Rule 1 Each bit position maps to exactly one physical output signal
Rule 2 Purely combinational — no registers in the output path
Rule 3 The mapping is defined per-wrapper — different for each application
Rule 4 Special outputs (interrupt return, timer, alarm) may use extra logic

Inside the Input Decoder — Priority Encoder

The priority encoder uses an IF / ELSIF / ELSIF chain — the first matching condition wins. This ensures only one event fires per clock cycle even if multiple inputs arrive simultaneously.

The fsm_busy gate wraps the entire encoder: when the FSM core is mid-pipeline, no new event is sent. This prevents the same physical event from being counted twice during the 2-cycle pipeline latency.

Interrupt signals (emergency, cancel, bad parity) skip the priority encoder and go directly to the interrupt_event port — they are always serviced immediately.

Inside the Output Decoder — Bit Mapping

Example: Traffic Light wrapper output decoding. Each bit in output_action drives exactly one physical pin. The decode is purely combinational — red_led <= output_action(0).

VHDL Code — Annotated Examples
Traffic Light — Input Decoder (with fsm_busy gate)
input_encode : PROCESS(clk)
BEGIN
  IF rising_edge(clk) THEN
    event_code <= (OTHERS => '0');        -- default: no event
    IF fsm_busy = '0' THEN               -- pipeline gate
      IF    pedestrian_btn = '1' THEN
        event_code <= EV_PEDESTRIAN_BTN; -- "0000000001"
      ELSIF car_sensor     = '1' THEN
        event_code <= EV_CAR_SENSOR;     -- "0000000010"
      ELSIF timer_done     = '1' THEN
        event_code <= EV_TIMER_DONE;     -- "0000000100"
      END IF;
    END IF;
  END IF;
END PROCESS;
The IF fsm_busy = '0' guard is the key safety mechanism — it ensures the 2-stage pipeline completes before a new event enters.
Traffic Light — Output Decoder (pure combinational)
output_decode : PROCESS(output_action, output_valid)
BEGIN
  red_led    <= '0'; yellow_led <= '0';
  green_led  <= '0'; ped_signal <= '0';
  timer_start <= '0';
  IF output_valid = '1' THEN
    red_led    <= output_action(0); -- bit 0 → red LED
    yellow_led <= output_action(1); -- bit 1 → yellow LED
    green_led  <= output_action(2); -- bit 2 → green LED
    ped_signal <= output_action(3); -- bit 3 → walk signal
    timer_start<= output_action(8); -- bit 8 → timer ctrl
  END IF;
END PROCESS;
Zero application logic — just bit assignments. Any change to FSM behavior is made in the ROM, not here.
Serial — Edge Detection (rx_valid rising edge only)
edge_detect : PROCESS(clk)
BEGIN
  IF rising_edge(clk) THEN
    rx_valid_prev <= rx_valid; -- remember last cycle
    tx_ready_prev <= tx_ready;
  END IF;
END PROCESS;

-- Rising edge = current HIGH, previous LOW
rx_valid_rise <= rx_valid AND NOT rx_valid_prev;
tx_ready_rise <= tx_ready AND NOT tx_ready_prev;
Without edge detection, a signal held HIGH for 3 cycles would fire 3 events through the pipeline — one per clock. Edge detection fires exactly once per physical signal assertion.
Vending — Interrupt Return Detection (cancel refund)
int_return : PROCESS(clk)
BEGIN
  IF rising_edge(clk) THEN
    state_prev <= state_code;  -- track previous state
    -- Detect COLLECT→IDLE transition (interrupt fired)
    IF state_prev = VM_COLLECT
       AND state_code = VM_IDLE THEN
      int_change_pulse <= '1'; -- assert refund for 1 cycle
    ELSE
      int_change_pulse <= '0';
    END IF;
  END IF;
END PROCESS;

change_return <= output_action(1) OR int_change_pulse;
The interrupt jump bypasses normal output_action. This process catches the forced state change and generates the refund pulse that the ROM path would have produced — ensuring coins are always returned.
Common Questions about Wrappers
Why not put the input/output logic directly in the FSM core?▼
The FSM core only speaks in abstract event codes and output bitmasks — it has no knowledge of traffic lights or vending machines. This separation is what makes the core reusable. If the core handled real signals, you'd need to rewrite or reconfigure it for every application, defeating the whole point.
Can a wrapper have state of its own?▼
Yes — some wrappers contain small auxiliary processes with registers (like the elevator's floor counter or the serial wrapper's shift register and parity accumulator). These are not FSM states — they are support logic that helps translate physical signals into events or that accumulates data across multiple state transitions. The state machine logic itself always lives in the generic core.
What if two inputs arrive simultaneously?▼
The priority encoder handles this. The IF/ELSIF chain means only the highest-priority condition fires. Lower-priority signals are effectively held for the next cycle (if the FSM is busy) or simply missed (if held for only one cycle). In real hardware design, the application spec defines which signal takes priority — e.g., emergency always beats everything else.
How does the wrapper know which ROM section to use?▼
Each wrapper sets config_id to a fixed constant — traffic light always drives "00", vending always drives "01", etc. This constant is connected directly to the FSM core's config_id input port and never changes during operation. The ROM uses this to select the correct section for every address lookup.

Configuration ROM

The lookup table that defines all FSM behavior — config_rom.vhd

The Configuration ROM is the brain of the entire system. It is a large lookup table that answers one question every clock cycle: "Given that I am in state X and event Y just happened, what should I do next?" The answer — encoded in a 32-bit word — tells the FSM core: which state to go to, which output signals to assert, and whether to start a timer.

Think of it like a spreadsheet with rows indexed by (controller, state, event) and columns for (next_state, outputs, flags). Changing behavior is as simple as editing a cell in the spreadsheet — no logic changes required.

Why is the Address Space 2¹⁷ = 131,072?

The ROM address is formed by concatenating three fields:

config_id 2 bits = 4 possible controllers (00, 01, 10, 11)
current_state 5 bits = up to 32 states per controller
event_code 10 bits = up to 1,024 distinguishable events
Total address width = 2 + 5 + 10 = 17 bits
2¹⁷ = 131,072 possible addresses

📐 In Practice: Sparse ROM

Each controller section has 215 = 32,768 addresses (5-bit state × 10-bit event). With 32-bit words, that is 128 KB per section. However, only 20–30 entries are non-zero per controller — all others default to x"00000000" (stay in state 0, no outputs). VHDL synthesis tools automatically infer efficient BRAM or LUT-RAM from this sparse structure. The OTHERS => (OTHERS => '0') default is intentional: an unmapped (state, event) pair safely does nothing.

Traffic Light FSM

traffic_light_wrapper.vhd · config_id = "00" · 6 states · 13 scenario steps

🚦 Problem Being Solved

A road intersection needs a controller that cycles Red→Green→Yellow on a timer, handles pedestrian button presses to queue a PED_CROSS phase, and supports an emergency override that blanks all lights for emergency vehicles. The FSM enforces correct ordering — you can never reach GREEN without passing through RED first.

Understanding States, Events & Outputs
States (what the FSM is doing)
S0 IDLE All lights off. Intersection inactive — no car or pedestrian detected. System idle after reset.
S1 RED Red light ON. All vehicles stopped. Either a car_sensor or pedestrian_btn triggered entry. Timer started.
S2 GREEN Green light ON. Vehicles may proceed. Entered after RED timer expires. Timer started.
S3 YELLOW Yellow light ON. Warning: prepare to stop. Entered after GREEN timer expires. Short timer.
S4 PED_WAIT Red ON, pedestrian request queued. Entered from RED or GREEN when pedestrian_btn fires.
S5 PED_CROSS Red ON + walk signal ON. Pedestrian actively crossing. Entered from PED_WAIT after timer.
Events (what triggers transitions)
pedestrian_btn (bit[0]=1) Pedestrian pressed the crossing button. Valid from IDLE, RED, GREEN.
car_sensor (bit[1]=2) Vehicle detected by loop sensor. Valid from IDLE only — triggers entry to RED.
timer_done (bit[2]=4) External timer expired. Drives all phase transitions: RED→GREEN, GREEN→YELLOW, YELLOW→RED, PED_WAIT→PED_CROSS, PED_CROSS→RED.
INTERRUPT (emergency) Emergency vehicle override. ROM interrupt_en entry forces FSM → IDLE in 1 cycle.
Outputs (physical signals asserted)
output_action[0] → red_led: HIGH in RED, PED_WAIT, PED_CROSS
output_action[1] → yellow_led: HIGH in YELLOW only
output_action[2] → green_led: HIGH in GREEN only
output_action[3] → ped_signal: HIGH in PED_CROSS only

vending_wrapper.vhd · config_id = "01" · 5 states · 16 scenario steps

🥤 Problem Being Solved

Elevator FSM

elevator_wrapper.vhd · config_id = "10" · 5 states · 21 scenario steps

🛗 Problem Being Solved

An elevator must travel between 5 floors, open its door safely upon arrival, detect and hold for door obstructions, handle overload (weight sensor), and respond to emergency stop. The FSM prevents movement while doors are open, and ROM HOLD entries guard against pipeline spurious re-entry events when transitioning into MOVE states.

Understanding States, Events & Outputs
States
S0 EL_IDLE Elevator parked. motor_up=0, motor_down=0, door=0, alarm=0. Waiting for floor request.
S1 EL_MOVE_UP motor_up=1. Floor counter incrementing. ROM HOLD entry guards against spurious go_up re-entry.
S2 EL_MOVE_DOWN motor_down=1. Floor counter decrementing. Same HOLD guard pattern as MOVE_UP.
S3 EL_DOOR_OPEN door=1. Boarding/alighting. door_sensor (obstruction) → HOLD in DOOR_OPEN. door_clear → DOOR_CLOSE.
S4 EL_DOOR_CLOSE door=0. Closing sequence. Auto-transitions to IDLE after one cycle.
Events & Outputs
go_up (bit[0]=1) Floor request above current floor. IDLE→MOVE_UP. HOLD in MOVE_UP (pipeline guard).
go_down (bit[1]=2) Floor request below current floor. IDLE→MOVE_DOWN.
arrived (bit[2]=4) current_floor = target_floor. MOVE_UP or MOVE_DOWN → DOOR_OPEN.
door_clear (bit[3]=8) No obstruction. DOOR_OPEN → DOOR_CLOSE.
door_sensor (bit[4]=16) Obstruction detected. HOLD in DOOR_OPEN (door stays open).
emergency_btn ⚠ INTERRUPT EL_INTERRUPT_EVENT. Forces FSM → IDLE. alarm_buzzer=1 asserted by wrapper.
Outputs
motor_up HIGH in EL_MOVE_UP · motor_down HIGH in EL_MOVE_DOWN
door_open HIGH in EL_DOOR_OPEN · alarm_buzzer HIGH on emergency interrupt

Floor Counter VHDL
elevator_wrapper.vhd — floor_counter process
floor_counter : process(clk)
begin
    if rising_edge(clk) then
        if    state_code = "00001" and current_floor < target_floor then
            current_floor <= current_floor + 1;  -- MOVE_UP
        elsif state_code = "00010" and current_floor > target_floor then
            current_floor <= current_floor - 1;  -- MOVE_DOWN
        end if;
        if current_floor = target_floor then
            arrived_event <= '1';               -- triggers FSM → DOOR_OPEN
        end if;
    end if;
end process;

Serial Communication FSM

serial_wrapper.vhd · config_id = "11" · 12 states · 20 scenario steps

📡 Why Serial Needs an FSM

Serial communication is inherently sequential: START → BIT0..BIT7 → STOP → COMPLETE — each bit must arrive in order. An FSM enforces this ordering with no possibility of skipping a bit. Each state corresponds to exactly one bit position. Parity checking happens in the STOP state — if it fails, SP_INTERRUPT_EVENT fires immediately, discarding the corrupt frame.

Understanding States, Events & Outputs
12 Frame States
S0 SP_IDLE Waiting for a new frame. rx_valid rising edge triggers → SP_START.
S1 SP_START Start bit detected. Next rx_valid → SP_RX_BIT0.
S2–S9 SP_BIT0..7 Each state receives exactly one data bit. rx_valid rising edge advances state. 8 sequential states.
S10 SP_STOP Stop bit. Parity checked here. Good parity + rx_valid → SP_COMPLETE. Bad parity → SP_INTERRUPT_EVENT fires.
S11 SP_COMPLETE Full frame received and validated. tx_enable='1'. tx_ready → SP_IDLE. Auto-loops if no tx_ready in 16 cycles (timeout).
Events & Outputs
rx_valid rising edge (bit[8]=256) Each rising edge advances the FSM one state. Wrapper uses edge detection (rx_valid AND NOT rx_valid_prev) to fire exactly once per bit.
tx_ready rising edge (bit[9]=512) COMPLETE → IDLE. Downstream device acknowledges byte.
BAD PARITY ⚠ INTERRUPT SP_INTERRUPT_EVENT="1100000000" fires at STOP if parity check fails. FSM → SP_IDLE. parity_err='1'.
Outputs
tx_enable HIGH in SP_COMPLETE — signals byte is ready to transmit downstream
parity_err HIGH when bad-parity interrupt fires — frame discarded
Byte 0xA5 = 1010 0101 Example
LSB-first: BIT0=1, BIT1=0, BIT2=1, BIT3=0, BIT4=0, BIT5=1, BIT6=0, BIT7=1
XOR parity: 1⊕0⊕1⊕0⊕0⊕1⊕0⊕1 = 0 → EVEN → parity_ok='1' → COMPLETE
Guided Walkthroughs — from Testbench Cases
Full Good Frame: SP_IDLE → START → BIT0–7 → STOP → COMPLETE → IDLE
1
rx_valid rising edge from IDLE: SP_INTERRUPT_EVENT checked — not matched. event_code=256 (bit[8]). SP_IDLE → SP_START(00001).
2
rx_valid ×8: Each rising edge advances one state. BIT0→BIT1→...→BIT7. Wrapper records each bit in shift register. 8 separate transitions, each through the 2-stage pipeline.
3
rx_valid at SP_BIT7 → SP_STOP(01010). Parity XOR computed: for 0xA5, parity=0. rx_parity_ok='1'.
4
rx_valid at SP_STOP with parity_ok='1': event_code=256 (good). → SP_COMPLETE(01011). tx_enable='1'.
5
tx_ready rising edge: event_code=512 (bit[9]). SP_COMPLETE → SP_IDLE. tx_enable='0'. Frame processing complete. Byte 0xA5 forwarded.
Bad Parity Error Path
1
Frame starts normally: IDLE → START → BIT0..7 as above. Bits received with odd parity (bit error injected).
2
rx_valid at SP_STOP: parity XOR ≠ 0. rx_parity_ok='0'. Wrapper sets event_code = SP_INTERRUPT_EVENT="1100000000" instead of normal rx_valid code.
3
Interrupt fires: FSM sees interrupt_event match + interrupt_en='1' in ROM. FSM → SP_IDLE in 1 pipeline cycle. parity_err='1' output asserted.
4
Frame discarded: tx_enable never asserts. Corrupt byte silently dropped. FSM immediately ready for next frame from SP_IDLE.

serial_wrapper.vhd — edge detect + parity check
input_encoder: PROCESS(rx_valid, rx_valid_prev, rx_parity_ok)
BEGIN
    event_code <= (OTHERS => '0');
    IF rx_valid = '1' AND rx_valid_prev = '0' THEN   -- rising edge only
        IF rx_parity_ok = '1' THEN
            event_code <= "0100000000";     -- bit8: good
        ELSE
            event_code <= SP_INTERRUPT_EVENT; -- bad parity → IDLE
        END IF;
    ELSIF tx_ready = '1' AND tx_ready_prev = '0' THEN
        event_code <= "1000000000";         -- bit9: tx_ready
    END IF;
END PROCESS;

What is GHDL?

GHDL is a free, open-source VHDL simulator. It compiles VHDL into a native binary, runs the simulation, and produces VCD waveform files. Use --std=08 for VHDL-2008 compliance.

🧪

What is a Testbench?

A VHDL file with no ports that instantiates your design, drives stimulus signals in a process, and checks outputs using assert. It is the VHDL equivalent of a unit test.

📊

GTKWave

Free waveform viewer that reads VCD files from GHDL. This panel now also renders VCD data directly so you can see real hardware timing in the browser without GTKWave.

📡

Backend Bridge

sim_server.py wraps GHDL with a Flask REST API + SSE stream. The frontend polls for results and renders cycle-by-cycle state, events, ROM addresses, and decoded outputs.

How to Read Digital Waveforms

clk

Square-wave clock. Count rising edges to measure latency in clock cycles. Purple segments in the waveform.

fsm_busy

Pulses HIGH for exactly 1 cycle during pipeline processing. Proves the 2-stage pipeline is functioning.

state_code

Changes 2 cycles after event_code fires — the pipeline latency. This is the primary correctness check.

output_action

16-bit bitmask decoded per wrapper: traffic=LEDs [3:0], elevator=motor bits, etc. Changes with state_code.

Key Takeaways
1Table-driven FSM separates behavior from structure. generic_fsm.vhd never changes; only the ROM data changes per application.
22-stage pipeline is essential. ROM lookup takes one full cycle; Stage 2 registers the data so Stage 3 can update state with correct setup timing.
3hold_state prevents silent resets. Without it, any unmapped event hits the OTHERS=>0 default and sends the FSM to IDLE unexpectedly.
4Interrupts provide safe emergency returns. Any application can define a "cancel everything" event without special FSM logic.
5Wrappers are thin adapters, not logic. Each wrapper only encodes inputs → event_code and decodes output_action → physical signals.
6Edge detection prevents double-sampling. Level-based events fire multiple times through a pipelined FSM; edge detection fires exactly once per physical event.
